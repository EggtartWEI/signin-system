## 1. 执行摘要 (Executive Summary)

### 产品概念

**统一认证中心 (Unified Auth Hub)** 是一个基于"OA 借壳认证 + JWT 无状态令牌"架构的企业级身份认证服务。该服务通过复用企业现有的 AWS OA 系统登录凭证，为所有内部业务应用提供统一、安全、合规的单点登录（SSO）解决方案。

### 核心问题

当前企业内部存在多个独立业务系统，每个系统都需要单独的认证机制。现有的 `attendance_login_only.py` 仅支持单一 OA 系统的简单登录，缺少：
- 审计日志功能（不满足等保2.0要求）
- 会话管理机制（存在安全风险）
- 异常检测和防护（如暴力破解）
- 可扩展的标准化接口（OAuth 2.0/OIDC）

### 目标市场

**主要市场**：
- 企业内部 IT 系统（5-50个业务应用）
- 需要统一认证的中大型企业（100-10000员工）
- 受到网络安全等级保护（等保2.0）合规要求的组织

**关键利益相关者**：
- 企业员工（最终用户）
- IT 开发团队（系统集成者）
- 安全审计部门（合规监管者）
- 系统管理员（运维人员）

### 价值主张

1. **安全性**：通过 RSA 加密传输、JWT 签名、审计日志等多重安全机制，显著提升认证安全水平
2. **合规性**：内置完整的审计功能，满足等保2.0、个人信息保护法等法规要求
3. **开发效率**：提供标准化 REST API，业务系统接入时间从数天缩短到数小时
4. **用户体验**：员工只需一套 OA 凭证即可访问所有授权系统，减少密码记忆负担
5. **成本效益**：复用现有 OA 系统，无需构建独立的用户数据库，降低建设和维护成本

## 2. 问题陈述 (Problem Statement)

### 当前状态与痛点

#### 痛点 1：缺乏统一的认证标准

**现状**：
- 每个 IT 系统独立实现认证逻辑，代码重复率高
- 不同系统的密码策略、会话管理机制不统一
- 新系统接入需要重新开发认证模块

**影响**：
- 开发效率低下，每个新系统需要 2-5 天实现认证功能
- 维护成本高，密码策略变更需要同步修改多个系统
- 安全风险大，某个系统的认证漏洞可能影响整体安全

#### 痛点 2：安全合规风险

**现状**：
- 现有 `attendance_login_only.py` 无审计日志功能
- 缺少登录失败次数限制，易受暴力破解攻击
- 无异常登录检测机制
- Token 管理不规范（无过期时间、无撤销机制）

**影响**：
- **无法通过等级保护认证**：根据等保2.0"安全审计"要求，必须记录所有用户登录行为
- **合规风险**：《网络安全法》要求采取技术措施监测、记录网络运行状态和网络安全事件
- **潜在损失**：一旦发生安全事件，无法追溯攻击路径，企业面临法律和声誉风险

#### 痛点 3：用户体验不佳

**现状**：
- 员工需要为每个系统记住不同的用户名和密码
- 跨系统访问需要重复登录
- 忘记密码需要分别在各系统重置

**影响**：
- IT 服务台密码重置请求频繁（估计每月 50+ 次）
- 员工工作效率降低（每天平均 3-5 次重复登录）
- 存在密码复用风险（员工在多个系统使用相同密码）

#### 痛点 4：可扩展性差

**现状**：
- 现有系统与 OA 系统紧耦合，难以支持其他身份源（如 LDAP、AD）
- 无标准化协议支持（如 OAuth 2.0、SAML）
- 单体架构，无法水平扩展

**影响**：
- 无法支持企业并购后的系统整合
- 无法与第三方 SaaS 服务集成
- 用户增长后性能瓶颈难以解决

### 现有方案的不足

**方案 1：继续使用现有系统**
- ❌ 无法满足合规要求
- ❌ 安全风险持续存在
- ❌ 维护成本高

**方案 2：商业单点登录产品（如 Keycloak）**
- ❌ 部署复杂，需要专门团队维护
- ❌ 与现有 OA 系统集成困难
- ❌ 成本高昂（软件许可 + 服务器 + 人力）

**方案 3：完全重建用户数据库**
- ❌ 需要员工重新注册和激活
- ❌ 数据同步复杂（与 HR 系统）
- ❌ 迁移成本高，用户抵触

### 为什么现在必须解决

**紧迫性因素**：
1. **合规压力**：企业需要在 2026 年 Q2 前通过等保二级认证
2. **业务增长**：预计未来 6 个月将新增 3-5 个业务系统
3. **安全事件**：行业内已发生多起因认证漏洞导致的数据泄露事件
4. **成本压力**：IT 团队维护多个认证逻辑的人力成本过高

## 3. 解决方案概述 (Proposed Solution)

### 核心概念

**统一认证中心**采用"借壳认证"（Pass-through Authentication）模式，即：
1. **不存储用户密码**：用户的身份凭证仍然存储在 OA 系统中
2. **按需验证**：当用户尝试登录时，认证中心实时调用 OA 系统的登录接口验证
3. **信息快照**：验证成功后，从 OA 系统的响应页面中提取用户基本信息（姓名、部门等）
4. **令牌颁发**：使用 JWT 标准格式颁发访问令牌，包含用户信息的快照
5. **无状态验证**：业务系统通过验证令牌签名来确认用户身份，无需查询数据库

### 关键差异点

#### 差异点 1：懒加载模式

**传统方式**：
```
用户登录 → 查询本地数据库 → 验证密码 → 返回用户信息
```

**我们的方式**：
```
用户登录 → 实时调用 OA 系统 → 提取用户信息 → 签发 JWT
```

**优势**：
- 无需维护用户数据库（降低数据泄露风险）
- 用户信息变更（如部门调整）在下次登录时自动更新
- 减少系统间数据同步复杂度

#### 差异点 2：完整的审计日志系统

**实现**：
- 所有认证事件（登录成功/失败、Token 签发、验证失败）均记录审计日志
- 日志包含：时间戳、用户 ID、IP 地址、设备指纹、结果、失败原因
- 日志独立存储，支持防篡改（数字签名或 WORM 存储）

**价值**：
- 满足等保2.0"安全审计"要求
- 支持安全事件溯源和分析
- 提供用户行为审计报表

#### 差异点 3：等保 2.0 合规设计

**符合的等保要求**：
- ✅ 身份鉴别：支持双因素认证（Phase 2）
- ✅ 访问控制：基于角色的访问控制（RBAC）
- ✅ 安全审计：完整的日志记录和防篡改
- ✅ 数据完整性：JWT 签名确保令牌不被篡改
- ✅ 数据保密性：全程 HTTPS + RSA 加密

#### 差异点 4：渐进式部署策略

**策略**：
- 不强制所有系统一次性迁移
- 新系统直接集成，老系统逐步改造
- 提供"并行运行"模式（新旧认证系统同时存在）

**优势**：
- 降低迁移风险
- 减少业务中断
- 逐步验证系统稳定性

### 为什么这个方案会成功

#### 成功因素 1：复用现有资产

- OA 系统已运行多年，稳定可靠
- 员工已熟悉 OA 登录流程，无需重新培训
- 无需构建新的用户管理后台

#### 成功因素 2：技术成熟度高

- FastAPI：现代 Python Web 框架，性能优秀
- JWT：行业标准（RFC 7519），生态系统完善
- OAuth 2.0/OIDC：业界标准协议，便于扩展

#### 成功因素 3：满足真实业务需求

- 直接解决合规问题（等保 2.0）
- 显著降低开发维护成本
- 提升用户体验（减少重复登录）

#### 成功因素 4：风险可控

- MVP 范围清晰，1-2 周可完成
- 有完整的应急预案（Plan B）
- 不影响现有系统运行

### 高层次愿景

**短期（3-6 个月）**：
- 成为所有新建业务系统的默认认证服务
- 通过等保二级认证
- 支持 5-10 个业务系统接入

**中期（6-12 个月）**：
- 支持 OAuth 2.0/OIDC 标准协议
- 支持多因素认证（MFA）
- 接入 20+ 业务系统，覆盖 80% 企业应用

**长期（1-2 年）**：
- 扩展为统一身份管理平台（IAM）
- 支持多种身份源（LDAP、AD、SAML、社交登录）
- 实现零信任架构核心组件

## 5. 目标与成功指标 (Goals & Success Metrics)

### 业务目标（SMART 原则）

#### 目标 1：系统接入数量

**具体目标**：
- **3 个月内**：支持 5 个业务系统接入（包括 2 个现有系统改造 + 3 个新建系统）
- **6 个月内**：支持 10 个业务系统接入
- **12 个月内**：支持 20 个业务系统接入，覆盖 80% 企业应用

**衡量方式**：
- 成功接入并运行的业务系统数量
- 每月新增接入系统数量

#### 目标 2：认证请求处理量

**具体目标**：
- **日均认证请求**：从 0 增长到 10,000 次/天
- **峰值处理能力**：支持 1,000 QPS（每秒查询数）

**衡量方式**：
- 日均 POST /auth/login 请求次数
- 日均 GET /auth/me (Token 验证) 请求次数
- 峰值时段的响应时间分布

#### 目标 3：系统可用性

**具体目标**：
- **月度可用性**：≥ 99.9%（每月停机时间 < 43.2 分钟）
- **年度可用性**：≥ 99.5%（年度停机时间 < 43.8 小时）

**衡量方式**：
- 系统正常运行时间 / 总时间
- 计划内停机（升级、维护）vs 计划外停机（故障）
- 平均恢复时间（MTTR）

#### 目标 4：成本节约

**具体目标**：
- **开发成本**：每个新系统接入时间从 3-5 天降低到 0.5 天（节省 80%）
- **维护成本**：减少 50% 的认证相关维护工作
- **合规成本**：避免因不合规导致的罚款（假设单次罚款 50-100 万）

**衡量方式**：
- 接入新系统的实际人天数
- 认证相关故障数量和修复时间
- 合规审计通过率

### 用户成功指标

#### 指标 1：登录成功率

**目标**：登录成功率 ≥ 99%

**计算方式**：
```
登录成功率 = (成功登录次数 / 总登录尝试次数) × 100%
```

**数据来源**：审计日志中 `login_success` 和 `login_failure` 事件

**基线**：当前 OA 系统登录成功率约 98%（主要因密码错误）

#### 指标 2：平均登录响应时间

**目标**：P50（中位数）< 500ms，P95 < 1000ms

**计算方式**：
- 测量从用户提交登录表单到返回 JWT Token 的时间
- 包含：OA 系统调用时间 + 用户信息提取时间 + JWT 签发时间

**数据来源**：应用性能监控（APM）

**基线**：当前 `attendance_login_only.py` 响应时间约 800-1500ms

#### 指标 3：用户满意度

**目标**：
- 用户投诉 < 1 次/月
- NPS（净推荐值）≥ 40

**衡量方式**：
- IT 服务台收到的认证相关投诉数量
- 季度用户满意度调查（1-5 分制）

#### 指标 4：密码重置请求减少

**目标**：密码重置请求减少 60%

**计算方式**：
```
减少比例 = (改造前月均重置次数 - 改造后月均重置次数) / 改造前月均重置次数 × 100%
```

**基线**：当前约 50 次/月

### 关键绩效指标 (KPI)

#### KPI 1：Token 验证成功率

**定义**：业务系统验证 JWT Token 时的成功率

**目标**：≥ 99.5%

**计算方式**：
```
验证成功率 = (验证成功次数 / (验证成功 + 验证失败次数)) × 100%
```

**失败原因分类**：
- Token 过期（正常）
- Token 签名无效（异常，可能是攻击）
- Token 格式错误（客户端集成问题）

#### KPI 2：安全事件响应时间

**定义**：从检测到安全异常到响应的时间

**目标**：
- 暴力破解检测响应：< 5 分钟（自动封锁 IP）
- 异地登录告警响应：< 30 分钟（人工介入）
- 系统故障响应：< 10 分钟（自动告警）

#### KPI 3：合规检查通过率

**目标**：等保 2.0 二级认证一次性通过率 100%

**衡量方式**：
- 预评估得分（自评估）
- 正式评估得分（第三方评估）

**关键控制点**：
- 身份鉴别：100 分
- 访问控制：≥ 90 分
- 安全审计：≥ 90 分
- 数据完整性：100 分

#### KPI 4：系统性能指标

**目标**：
- CPU 使用率：< 70%（正常负载）
- 内存使用率：< 80%
- 响应时间 P95：< 1000ms

#### 功能 2：JWT 令牌服务 (Token Service)

**需求描述**：
实现 JWT 的签发、验证、刷新功能，遵循 RFC 7519 标准。

**具体实现**：
- [ ] `create_token(user_info)` - 签发访问令牌（Access Token）
- [ ] `verify_token(token)` - 验证令牌签名和有效期
- [ ] `refresh_token(refresh_token)` - 签发新的访问令牌（可选，MVP 阶段可简化）

**Token 载荷结构**：
```json
{
  "sub": "12061413",           // 用户 ID (Subject)
  "name": "李茗",               // 姓名
  "dept": "人工智能工作室",     // 部门 (快照)
  "iat": 1700000000,           // 签发时间 (Issued At)
  "exp": 1700038400            // 过期时间 (Expiration, 8小时后)
}
```

**验收标准**：
- Token 使用 HS256 或 RS256 算法签名
- 验证失败时返回明确的错误信息（过期、签名无效、格式错误）
- 签名密钥通过环境变量管理

**预估工作量**：0.5 天

## 7. MVP 后愿景 (Post-MVP Vision)

### Phase 2 功能规划（发布后 3-6 个月）

#### 功能 1：多因素认证（MFA）

**需求背景**：
等保 2.0 三级要求"身份鉴别"采用两种或两种以上组合的鉴别技术。虽然 MVP 阶段满足二级要求，但为了提升安全性，需要添加 MFA。

**实现方案**：
- **TOTP (Time-based One-Time Password)**：
  - 用户使用 Google Authenticator、Microsoft Authenticator 等应用
  - 登录时输入密码 + 6 位动态验证码
- **短信验证码**：
  - 向用户手机发送 6 位验证码
  - 有效期 5 分钟

**技术实现**：
```python
# TOTP 伪代码
def verify_mfa(username, code):
    secret = get_user_mfa_secret(username)
    totp = pyotp.TOTP(secret)
    return totp.verify(code, valid_window=1)  # 允许时间偏差±1个时间步长
```

**验收标准**：
- MFA 验码正确率 > 99.5%
- 验证码响应时间 < 500ms
- 用户可选择启用/禁用 MFA

**预估工作量**：5 天

---

#### 功能 2：异常登录检测

**需求背景**：
防止账号被盗用后的异常行为，如异地登录、异常时间登录等。

**检测规则**：
1. **异地登录检测**：
   - 记录用户常用的登录地点（IP 地址地理定位）
   - 当登录地点与常用地点相差 > 500km 时，触发告警

2. **异常时间检测**：
   - 检测非工作时间的登录（如凌晨 2-6 点）
   - 检测短时间内跨地域登录（不可能的旅行速度）

3. **设备指纹检测**：
   - 记录用户常用设备（User-Agent 浏览器指纹）
   - 新设备登录时要求额外验证

**告警机制**：
- 实时告警：发送邮件/短信给用户
- 管理员通知：高危异常（如连续 3 次异地登录）通知安全团队
- 自动响应：暂时锁定账号，强制重置密码

**验收标准**：
- 误报率 < 5%
- 检测响应时间 < 1 秒
- 告警送达率 > 99%

**预估工作量**：3 天

---

#### 功能 3：审计报表系统

**需求背景**：
安全审计人员和系统管理员需要可视化的审计报表，而不是查看原始日志。

**报表类型**：

1. **用户活跃度报表**：
   ```
   - 日/周/月活跃用户数（DAU/WAU/MAU）
   - 部门登录分布（饼图）
   - 用户登录热力图（按时间段）
   - 长期未登录用户列表（可能离职）
   ```

2. **安全事件报表**：
   ```
   - 暴力破解尝试统计（按 IP、时间段）
   - 异地登录事件列表（地图展示）
   - 登录失败 Top 10 用户（可能存在攻击）
   - OA 系统故障次数统计
   ```

3. **系统性能报表**：
   ```
   - 登录响应时间趋势（P50/P95/P99）
   - Token 验证成功率
   - 系统可用性（SLA）
   - 并发请求数统计
   ```

**实现方案**：
- 后端：Python 数据分析（Pandas）
- 前端：轻量级图表库（Chart.js 或 ECharts）
- 导出：支持 PDF、Excel 格式

**验收标准**：
- 报表生成时间 < 10 秒（100 万条日志）
- 支持自定义时间范围查询
- 报表可定时发送邮件（如每周一上午 9 点）

**预估工作量**：5 天

#### 功能 4：OAuth 2.0 / OIDC 完整实现

**需求背景**：
支持标准化授权协议，便于与企业级应用（如 Jira、Confluence、GitLab）集成。

**实现功能**：
- **授权码流程**（Authorization Code Flow）
- **PKCE 扩展**（Proof Key for Code Exchange，增强安全性）
- **Token 刷新**（Refresh Token）
- **标准端点**：
  - `/authorize` - 授权端点
  - `/token` - 令牌端点
  - `/userinfo` - 用户信息端点
  - `/.well-known/openid-configuration` - 发现文档

**OIDC (OpenID Connect) 扩展**：
- ID Token（包含用户身份信息）
- 标准Claims（sub、name、email、picture 等）

**验收标准**：
- 通过 [OAuth 2.0 Playground](https://www.oauth.com/playground/) 测试

- 与 3 个第三方应用（如 Jira、GitLab）成功集成

**预估工作量**：10 天
